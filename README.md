# Запуск
Запуск программы происходит через `docker compose up -d`. Перед этим, нужно создать .env файл со значениями, представленными в .env.exmaple.

Если по какой-то причине не удалось запустить программу через docker cmopose, то вот инструкция для ручного запуска:  
1. Оставить в docker compose только сервис с postgres - остальные удалить;
2. `go install github.com/pressly/goose/v3/cmd/goose@3.26.0`
3. `docker-compose up -d`
4. `export GOOSE_DRIVER=postgres`
5. `export GOOSE_DBSTRING=postgres://${DB_USER}:${DB_PASSWORD}@postgres:${DB_PORT}/${DB_NAME}?sslmode=disable`
6. `cd ./migrations`
7. `goose up`
8. `go run cmd/pr-service/main.go`

# Архитектурные и неочень решения
Разместил файлы с dto возле контроллеров, я думаю, пусть такие структуры будут ближе к тому месту, где они используются.  
Хотя думал также над вариантом папки dto в internal, где будут храниться все dto.

Изменил текст сообщений возвращаемых ошибок с resource not found на более информативные: team not found и т.п.

Обернул функции контроллера в роутинге в собственный тип функций для централизированной оработки ошибок.

При вызове ручки /users/getReview добавил логику проверки наличия указанного пользователя.  
Возвращает 404 если пользователь не найден. Мне кажется, в данном процессе нужна такая проверка, было бы странно получить успешный ответ при отсутствии пользователя.

Оставил возможным добавление ПР-а при отсутствии свободного ревьюера. Мне кажется, по бизнес-логике должна быть возможность добавления ревьюера позже: после освобождения или какая-нибудь задача по расписанию, которая будет проверять пустые ПР-ы и добавлять к ним ревьюеров, если есть свободный.

Покрыл тестами сервис ПР-ов, остальные сервисы просто делегируют работу репозиториям, думаю, нет смысла их покрывать.
